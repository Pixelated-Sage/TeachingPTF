You are setting up production deployment for the Live Coding Classroom platform. Frontend deploys to Vercel; backend runs persistently on the instructor's own Linux PC, exposed via Cloudflare Tunnel. This is for a small initial test (2-3 students first) before scaling to the full class — deployment must be stable enough to trust but doesn't need cloud-scale infrastructure.

---

## PART 1 — Backend Resilience (Linux PC, "must not close")

- Install and configure **PM2** to run the Express/Socket.io backend process. PM2 must:
  - Auto-restart the process if it crashes
  - Keep running in the background (not tied to an open terminal session)
  - Provide accessible logs (`pm2 logs`) for debugging during test sessions
- Set up a **systemd service** (or PM2's own `pm2 startup` + `pm2 save` mechanism) so the backend automatically starts on PC boot/reboot without manual intervention.
- Do NOT containerize with Docker — for a single-PC deployment with no multi-instance requirement, this adds complexity without solving a real problem here. Revisit only if this ever moves to actual cloud/multi-instance hosting later.

## PART 2 — Public Exposure (Cloudflare Tunnel)

- Set up Cloudflare Tunnel pointing to the local backend (Express + Socket.io, likely port 5000 or whatever is currently configured).
- Confirm WebSocket traffic passes through the tunnel correctly — test with an actual live Socket.io connection, not just a basic HTTP request, since WebSocket behavior through tunnels can differ from plain HTTP.
- Document the resulting stable public URL/subdomain so the Vercel frontend's environment config can point to it.
- Update CORS configuration on the backend to allow the actual Vercel frontend origin (not `*` or a dev-only localhost origin).

## PART 3 — Frontend Deployment (Vercel)

- Deploy the Next.js frontend to Vercel.
- Configure environment variables pointing to the Cloudflare Tunnel backend URL (not localhost).
- Confirm the COOP/COEP headers (previously configured for WebContainer support) are correctly set in Vercel's production config (`vercel.json` or `next.config.ts` headers function) — re-verify this works in the actual deployed environment, not just local dev, since headers behave differently in Vercel's production serving versus local dev server injection.

## PART 4 — Access Model Confirmation (no new logic, just verify)

- Confirm existing `live_session_active` flag correctly gates the Live Classroom experience — students can log in and reach their dashboard anytime the backend is running, but Live Classroom mode specifically requires the instructor's "Go Live" toggle to be on.
- Confirm Assignments (if scheduled/active) remain accessible to targeted students independent of live status — this should already be the case, just verify no accidental coupling exists between the two systems.

## PART 5 — New Feature: Download Code as ZIP

- Add a "Download My Code" button in the student workspace.
- On click, bundle the student's current file tree (pulled from the database-persisted source of truth, not just the ephemeral WebContainer session) into a `.zip` file and trigger a browser download.
- This should work regardless of whether a live session is active — a student should be able to grab their own code at any time.
