# Backend Optimization Pass — Agent Tracking Doc

Use this doc as a checklist. Mark each item done/blocked as you complete it, and report deviations inline. This pass hardens the backend for 60-concurrent-student scale without introducing infrastructure you don't need yet (no Redis, no Docker, no multi-process — single Node process with in-memory caching is the correct scope for now).

---

## 1. Connection Pooling — Check This First

- [ ] Confirm whether the current Supabase connection string routes through **Supavisor** (Supabase's built-in connection pooler) or connects directly to Postgres.
- [ ] If not already using it, switch the connection string to the pooler endpoint (Supabase dashboard → Database settings → Connection Pooling).
- [ ] Re-test basic connectivity after switching before proceeding to other items — this alone may reduce pressure on the `max: 10` pool limit.

## 2. Session Token Validation Cache (highest priority — highest frequency hit)

- [ ] Add an in-memory cache (plain JS `Map` is sufficient at this scale — no Redis needed) mapping `session_token → { userId, expiresAt }`.
- [ ] On login, populate this cache alongside the existing DB write.
- [ ] On every authenticated request, check the in-memory cache FIRST. Only fall through to a DB query if the token isn't in cache (e.g. after a server restart).
- [ ] Respect the existing token expiry (24hr, from earlier hardening pass) — expired cache entries should be treated as invalid and fall through to DB re-check.
- [ ] This should eliminate the majority of per-request DB round-trips across the whole app.

## 3. Notes / Classroom Status / Assignment Definitions Cache

- [ ] Cache Notes content per classroom in memory, keyed by `classroomId`.
- [ ] Cache `live_session_active` / classroom status per classroom in memory.
- [ ] Cache published Assignment definitions (questions, timers, targeting) per assignment in memory.
- [ ] Invalidate/update each cache **exactly when its corresponding socket event already fires** — do not add polling or TTL-based expiry:
  - `classroom:notes_updated` → refresh Notes cache for that classroom
  - `classroom:live_status` → refresh status cache for that classroom
  - Assignment publish/edit action → refresh that assignment's cache entry
- [ ] Reads (student content fetch, assignment fetch) should hit these caches first; only miss to DB on cache-empty (e.g. after restart).

## 4. Mishap Log In-Memory Aggregate

- [ ] Maintain an in-memory structure: `{ classroomId: { studentId: { tab_switch: N, inactivity: N, paste_attempt: N, lastEventAt: timestamp } } }`.
- [ ] On receiving `mishap:batch` from a client, update this in-memory aggregate immediately (increment counts) BEFORE/independent of writing to DB.
- [ ] Raw individual mishap events still get written to `MishapLogs` table for durability/audit/future-analysis use — but on a **batched/delayed schedule** (see item 6), not synchronously per event.
- [ ] Admin's "Refresh" action on Observation Cards and Student Profile mishap summaries should read from this in-memory aggregate, NOT re-query/re-count raw DB rows.

## 5. Workspace Autosave — Write Reduction (not caching, a write-frequency fix)

- [ ] Before writing an autosave to `StudentWorkspaces`, diff the incoming file content against the last-saved version (in-memory comparison, e.g. hash or direct string compare) for that student.
- [ ] Only perform the DB write if content has actually changed since the last save — skip the write entirely if unchanged, even if the autosave timer fired.
- [ ] This does not change the autosave trigger interval itself — just adds a "did anything change" gate before the write happens.

## 6. Batched Database Writes for High-Frequency Events

- [ ] For mishap events specifically (highest-frequency, most likely to burst simultaneously — e.g. 60 students tab-switching near-simultaneously during a proctored moment): buffer incoming events in memory and flush to the database in a batch every few seconds (e.g. every 3-5 seconds) rather than one `INSERT` per event as it arrives.
- [ ] This reduces write pressure on the connection pool specifically during burst moments, while the in-memory aggregate (item 4) still gives admin instant real-time counts regardless of when the batched DB write actually happens.

## 7. Database Indexes

- [ ] Add indexes on foreign key columns used in joins/filters across the schema, at minimum: `student_id`, `classroom_id`, `question_id` on every table that has them (`Submissions`, `MishapLogs`, `TestSubmissions`, `AssignmentSubmissions`, `UserClassrooms`, etc.).
- [ ] Pay particular attention to the columns used in the admin aggregation query (`/api/admin/classroom/:id/details`) since that single query joins across roster, mishaps, quick questions, assignments, and notes — confirm all join columns involved are indexed.

## 8. Process Longevity Hygiene (small, cheap insurance)

- [ ] Configure PM2 for a scheduled restart (e.g. once daily during an off-hours window) to prevent slow in-memory growth from long uptime.
- [ ] On a classroom session ending (Go Live → End Live, or a Test/Assignment closing), clear that classroom's in-memory caches (notes, status, mishap aggregate) rather than letting them persist indefinitely — they'll simply repopulate on next access.

---

## Explicitly Out of Scope for This Pass
- No Redis, no external cache service — in-memory (single Node process) is correct at this scale.
- No Docker.
- No multi-process/load-balanced backend architecture — this is all single-PC, single-process optimization.
- No changes to autosave trigger interval/timing — only the write-skip-if-unchanged gate (item 5).

## Testing Before Sign-Off
- [ ] Simulate 20-30 concurrent fake student sessions hitting login → content fetch → workspace load together, confirm reduced DB query volume compared to pre-optimization (rough before/after comparison is enough, doesn't need to be scientific).
- [ ] Simulate a mishap-event burst (e.g. 20+ simultaneous `mishap:batch` emissions) and confirm admin's Observation Cards still reflect accurate counts without database read lag.
- [ ] Confirm workspace autosave skips writes correctly when content is unchanged (verify via logs or DB row `updated_at` not changing when no edits were made).
- [ ] Confirm cache correctly repopulates after a manual backend restart (i.e. nothing breaks on cold cache, it just falls through to DB once and re-caches).

Report back against this checklist item by item, flag anything that couldn't be completed as specified, and note any deviations with reasoning.
